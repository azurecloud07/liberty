1.Use Case Description

New Entity creation request is initiated by client or by client appointed outside counsel (typically a law firm), on IRIS, or by Intertrust entity admin personnel on NEC portal, as a service request.

 

This use case does not include request for adding new management and other owner/controllers for the new entity to be incorporated as that would be handled by the Fastlane 2.0 Identity KYC work stream. NEC will be responsible for capturing “target contact” information for identity KYC and passing that to the identity KYC work stream to request and collect pertinent data and documents. NEC will also capture risk assessments, entity relationship KYC data and data for transaction profile (TP) setup.

 

This use case limits to new entity incorporation requests by existing clients (concept of old-new). This essentially means that IRIS and NEC will have the client related information already. It is possible that at this time of new entity incorporation, there is material change in the client information since the time the client was incepted, which might necessitate additional KYC and risk assessment, but is not in scope of this use case. In addition, the scope of this use case does not extend to foreign registrations of a legal entity in addition to the domicile (for e.g. incorporation in US and foreign registration in another US state or in Cayman).

 

[For details on notifications and status updates refer use case NEC_22]

 

 

 

 

 

 

1.Actors

Primary Actor – Client

Primary Actor – Outside Counsel

Secondary Actor – Intertrust entity admin group user (“Intertrust User”)

 

 

1.Triggers

1.New Entity Creation request placed by Client or Outside Counsel on IRIS portal; or

2.New Entity Creation request placed on behalf of Client by Intertrust User on EDM. This is a trigger when the Client is unable to or unwilling to create the request on IRIS and would like to communicate the information on the new entity to be incorporated via email or a phone to the Intertrust User.

 

1.Preconditions

1.Client / Outside Counsel has access to IRIS system

2.Client / Outside Counsel with access to IRIS system is authorized to create new entity creation service request (write permission on IRIS)

3.Intertrust User has access to the EDM system

4.Intertrust User with access to EDM System is authorized to create new entity creation service requests (

5.Entity operations documents are available to be uploaded with the service request which will be required for relationship KYC and administration

6.Entity related information required for relationship KYC and risk assessment are available with the service request creator

7.Entity related information required to update all the government filing forms for the entity’s jurisdiction and the entity type is available with the service request creator

8.Filing forms and its underlying fields are configured in IRIS and NEC

9.Based on the jurisdiction, entity type and service line, the fields to be available on NEC and IRIS service request web forms are configured. Ideally, the fields in IRIS will be a subset of the fields for the NEC web form

10.All web form fields, both for IRIS and NEC, are mapped to the filing form’s underlying fields

11.Client is activated on the IRIS portal

12.Based on jurisdiction, process flow settings (acceptable signature formats, notary applicability, CAP meeting applicability etc. – use case NEC_05) are already configured

 

 

 

 

 

 

 

1.Basic Course of Events

1.IRIS user logs in to the IRIS system

2.IRIS user selects the option (e.g. link/button) to request for the creation of a new legal entity

3.Client related data is displayed and is read only/not editable (greyed out)

4.Option to create new entity is available.

5.List of existing entities for the specific client is displayed (single select dropdown), so that an entity can be cloned (separate use case)

6.IRIS user selects the option “single entity creation request”

7.IRIS user selects the jurisdiction from the options presented on the screen (single select dropdown)

8.IRIS system displays the types of entity that are applicable for the selected jurisdiction (single select dropdown)

9.IRIS user selects the entity type from the options presented on the screen

10.IRIS system displays the fields applicable to the jurisdiction and entity type selected by the user

11.Based on jurisdiction and entity type, the data and documents pertaining to entity relationship KYC which need to be provided for the new legal entity which is to be incorporated is made available

12.Option to upload completed, signed and scanned filing forms in PDF format for the new entity creation specific to the jurisdiction and entity type settings (separa te use case). As a next phase item, once uploaded, the forms are read through Optical Character Reader (OCR) and all relevant fields are extracted into the standard web form for review and edit (if required)

13.Fields displayed are a combination of mandatory / non-mandatory, dropdowns (master client data from underlying systems like ViewPoint), date and free form text fields

14.Option to add one or multiple, new or existing, owner-controllers is available.

a.Option to add from existing Intertrust or external owner-controllers who have already undergone Identity KYC by Intertrust is available (single select dropdown)

b.If user chooses the option to add a new owner-controller, fields to capture contact for the target who will provide pertinent data and documentation required for identity KYC will be displayed and are mandatory to be filled.

c.Details of a separate target contact for every new owner-controller for the new entity will be asked for. Option to use the same target contact for each new owner-controller should also be present to avoid having the client or outside counsel user fill the same data multiple times if the same target contact applies to all new owner controllers

 

 

a. 

1.The fields and data elements are shown in six sections to the user on the screen:

a.Client information (prefilled and read only/not editable)

b.Entity relationship KYC data (by jurisdiction, entity type and service line)

c.Target contact details for Individual KYC for every new owner-controller

d.Filing form data (by jurisdiction and entity type)

e.Incremental data for transaction profile

f.Operational documents (by jurisdiction and entity type)

1.Option to save data in multiple login sessions (save, save and exit, submit options). No data should be lost due to timeout or intentional logout.

2.User updates all the fields (at least the mandatory fields)

3.User uploads the relevant operational documents

4.User saves the updated service request form for the entity creation

5.User submits the form for entity creation by clicking on submit button. Clicking on submit button will submit the new entity creation service request in the IRIS system OR EDM System respectively.

6.User can exit the request creation form by clicking on the exit button. User will be asked to confirm if she/he wants to exit the page. Exiting will also save the data entered by the user.

7.END OF USE CASE

1.Post Conditions (Basic)

1.Service request to initiate new entity created in IRIS followed by creation on NEC, or initiated in NEC followed by creation on IRIS, when service request is created by IRIS Authorized Users (client and outside counsel personnel) or NEC Authorized Users (Intertrust entity admin team personnel)

2.Irrespective of the origin system of the service request, the service request ID is created in NEC  and the same ID is used by IRIS. Once the ID is created in NEC and the Client Data Service (CDS, the data layer for IRIS and Microsoft Interact, the CRM tool) has it, the same service request ID is shared with Interact, which initiates the sales revenue capture process on Interact for this new entity incorporation and administration

3.Email alert is sent to IRIS Authorized and NEC Authorized Users

4.Notification on NEC to authorized Intertrust internal users, for the Client, when service request created on IRIS

5.Notification on IRIS for Client and Outside Counsel when service request created on NEC

6.At this point the status of the service request will be ”Service request for new entity incorporation has been initiated”

7.Landing page for the specific user will be displayed

8.Once submitted, the request will not be editable by IRIS Authorized Users irrespective of whether the service request originated in IRIS or NEC

9.NEC Authorized Users can now enrich / edit the request with additional information with extensive audit trail of changes made

10.The target contact information is shared with the identity KYC module. Note that in case of multiple owner-controllers for whom identity KYC needs to be performed, the same number of separate identity KYC requests are handed off to the identity KYC module

11.Risk assessment entity relationship KYC data will be downloadable at the click of a button for the purpose of manual entity relationship KYC process. The data can be downloaded from the NEC by either the entity admin team personnel or another authenticated Intertrust compliance user, performing entity relationship KYC, to use the data for the manual Risk assessment entity relationship KYC. If NEC is integrated with the new risk engine, the handoff of the entity relationship KYC data to the risk engine for entity relationship risk assessment, will be automatic and not manual

12.The incremental data collected for transaction profile, in addition to the rest of the data required for creation of the profile and captured as a part of the other sections already, will be automatically handed off to the Transaction Profile (TP) and Transaction Monitoring (TM) module to create the initial skeletal version of the transaction profile with minimal data.

13.Operational documents, used eventually for entity administration purposes, will be saved in NEC until the time the entity is incorporated along with any other related governance documents to Laserfiche manually. The functional setup on operational documents (use case NEC_05) for the given jurisdiction and entity type will keep alerting the IRIS and NEC users by email and system notification till all expected operational documents are available and uploaded

 

1.Alternative Path 1

1.At any moment in the basic flow, the User saves incomplete form by clicking on save button and exits in IRIS portal.System stores draft form that can be accessed by the same user at any time.

2.Use case ends

1.Post Conditions (Alternative path 1)

1.Data is saved in the IRIS system

2.When the user accesses the form again, the user will be shown the section that was last saved by the user.

3.All the data saved earlier will be displayed in the IRIS system

4.No mention of the new service request yet on NEC

 

1.Alternative Path 2

1.User saves incomplete form by clicking on save button and exits in EDM System.

1.Post Conditions (Alternative path 2)

1.Data is saved in the EDM System

2.When the user accesses the form again, the user will be shown the section that was last saved by the user.

3.All the data saved earlier will be displayed in the EDM System

4.No mention of the new service request yet on IRIS
