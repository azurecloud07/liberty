Use Case 2

 

 

 

Introduction

Workstream

New Entity Creation (NEC)

Use Case Name

Request New Entity Creation using prefilled/signed PDF documents

Stakeholder(s)

a.Client

b.Outside counsel

c.Intertrust entity admin team user initiating new entity creation on NEC New Entity Creation portal

d.Interact (CRM portal)

References

Business requirements discussions with Intertrust

Traceability

NEC_02

1.Use Case Description

New Entity creation request is initiated by Client or by client appointed Outside Counsel (typically a law firm) on IRIS, or by Intertrust User on NEC portal, as a service request. The Client or the Outside Counsel will upload the completed, signed and scanned forms in .PDF format into IRIS. Alternatively, the request can be sent over phone or email (web forms on IRIS will not be used to submit data and documents) to Intertrust User, who will upload the completed, signed and scanned forms in .PDF format into NEC.

 

This use case does not include a request for adding new management and other owner/controllers for the new entity to be incorporated as that would be handled by the Fastlane 2.0 Identity KYC work stream. NEC will be responsible for capturing “target contact” information for identity KYC and passing that to the identity KYC work stream to request and collect pertinent data and documents. NEC will also capture risk assessment entity relationship KYC data and data for transaction profile (TP) setup.

 

Also, this use case limits to new entity incorporation requests by existing clients (concept of old-new). This essentially means that IRIS and NEC will have the client related information already. It is possible that at this time of new entity incorporation, there is material change in the client information since the time the client was incepted, which might necessitate additional KYC and risk assessment, but is not in scope.

In addition, the scope of this use case does not extend to registration of a legal entity outside the jurisdiction/domicile (for e.g. incorporation in US-Delaware jurisdiction, but registration, possibly for tax saving purposes, in US-Florida).

 

[For details on notifications and status updates refer use case NEC_22]

 

1.Actors

Primary Actor – Intertrust entity admin group admin (“Intertrust User”)

Secondary Actor – Client

Secondary Actor – Outside Counsel

1.Triggers

1.New Entity Creation request placed by Client or Outside Counsel on IRIS portal. Completed, signed and scanned, jurisdiction and legal entity specific filing forms in PDF format are uploaded; or

2.New Entity Creation request placed on behalf of Client by Intertrust User in NEC. Completed, signed and scanned, jurisdiction and legal entity specific filing forms in PDF format are uploaded.

 

1.Preconditions

1.Client / Outside Counsel User has access to IRIS system

2.Client / Outside Counsel User with access to IRIS system is authorized to create new entity creation service request for the specific client (write permission on IRIS).

3.Intertrust User has access to NEC system

4.Intertrust User with access to NEC system is authorized to create new entity creation request for the specific Client (write permission on IRIS).

5.Entity operational documents are available to be uploaded with the service request which will be required for relationship KYC and administration

6.Entity related information required for relationship KYC and risk assessment are available with the service request creator

7.All relevant government forms for the new entity incorporation request are filled and are signed by the respective authorized signatories.

8.The completed and signed forms are converted into PDF format

9.Fields in IRIS system are mapped to all the appropriate fields in NEC system

10.Fields in each of the filing forms are mapped correctly to the fields in the IRIS and NEC systems

11.Client information is already available on IRIS and NEC as this use case covers existing clients ONLY

12.Based on jurisdiction, process flow settings (applicable risk approval groups, acceptable signature formats, notary applicability, CAP meeting applicability) are already configured

 

 

1.Basic Course of Events

1.IRIS Authorized User logs in to the IRIS system

2.IRIS Authorized User selects the option (e.g. link/button) to create new entity creation Service Request.

3.Client related data is displayed and is not editable (greyed out)

4.Option to create new entity is available.

5.IRIS User makes a choice of jurisdiction, entity type (pertaining to the chosen jurisdiction) and service line

6.Option to create new entities by filling up web forms with pertinent jurisdiction, entity type and business line specific data is available (separate use case)

7.List of existing entities for the specific Client is displayed (single select dropdown), so that an entity can be cloned (Separate use case)

8.IRIS Authorized User selects option to create new entity creation request by uploading prefilled and signed pdf documents.

9.Based on jurisdiction and entity type, the data and documents pertaining to entity relationship KYC which need to be provided for the new legal entity which is to be incorporated is made available

** Mapping for data and documents to be sourced for relationship KYC – From Intertrust **

1.Based on predefined mapping, system displays list of filing forms that need to be completed, signed and submitted to the respective jurisdiction specific government incorporation office.

2.System displays upload button, to upload the selected PDF forms under the respective document names on this list.

3.With every form upload, option to add free form comment

4.One, many or all forms pertinent to the jurisdiction and entity type can be uploaded at a time

5.If the forms had been previously shared with the Intertrust User by the Client or the Outside Counsel, the Intertrust User can upload the documents using similar upload features on NEC portal (similar to the functionality available on IRIS, as determined by the jurisdiction and entity type)

6.Once submitted on IRIS or on NEC, the forms are available on NEC to be read in their native PDF format, or as a phase II enhancement using an integrated Optical Character Reader (OCR) tool.

7.Once read in NEC using the OCR tool, all the fields as per the selected government form are populated with the data as provided in the pre-filled PDF form

8.Fields which are not clearly read by the OCR are left blank and highlighted for manual entry.

9.All other entity related prefilled fields are editable

10.Option to add one or multiple owner-controllers is available

11.Option to add from existing Intertrust or external directors who have already undergone identity KYC by Intertrust is available (single select dropdown)

12.Alternatively, option to add a new owner-controller is available.

a.If user chooses the option to add a new owner-controller, fields to capture contact for the target who will provide pertinent data and documentation required for identity KYC will be displayed and are mandatory to be filled.

b.Details of a separate target contact for every new owner-controller for the new entity will be asked for. Option to use the same target contact for each new owner-controller should also be present to avoid having the client or outside counsel user fill the same data multiple times if the same target contact applies to all new owner controllers

1.The fields and data elements are in six sections to the NEC Authenticated User:

a.Client information (prefilled and not editable)

b.Entity relationship KYC data (by jurisdiction, entity type and service line)

c.Target contact details for Individual KYC for every new owner-controller

d.Filing form data (by jurisdiction and entity type)

e.Incremental data for transaction profile

f.Operational documents (by jurisdiction and entity type)

1.Option to save data (save, save and exit, submit options). No data once extracted using OCR from the PDF documents will be lost due to timeout or intentional logout.

2.NEC Authorized User updates all the fields (at least mandatory fields)

3.NEC Authorized User  uploads the relevant operational documents

4.NEC Authorized User  saves the updated form for entity creation

5.NEC Authorized User submits the form for entity creation by clicking on submit button. Clicking on submit button will submit the new entity creation request in the NEC system.

6.NEC Authorized User can exit the request creation form by clicking on the exit button. User will be asked to confirm if she/he wants to exit the page. Exiting will also save the data entered by the user

1.Post Conditions (Basic)

1.Service request to create new entity created in IRIS followed by creation on NEC, OR created in NEC followed by creation on IRIS, when service request is created by IRIS authorized users (client and outside counsel user) OR NEC authorized users (Intertrust entity admin team user), respectively

2.Irrespective of the application of origin of the service request (IRIS or NEC), the ID is always created in NEC and the same ID is used by IRIS. Once the ID is created in NEC and the Client Data Service (CDS, the data layer for IRIS and Microsoft Interact, the CRM tool) has it, the same service request ID is shared with Interact, which initiates the sales revenue capture process on Interact for this new entity incorporation and administration

Note : the service request reference will always be in the following format : SRxxxxx

It is a requirement that the reference stays the same in the 3 applications : IRIS, EDM and Interact.

1.Email alert is sent to IRIS Authorized and NEC Authorized Users

2.Notification on NEC for Intertrust Users for the Client, when service request created on IRIS

3.Notification on IRIS for IRIS Authorized Users when service request created on NEC

4.At this point the status of the service request will be ‘Initiated’

5.Once submitted, the request will not be editable by IRIS Authorized Users irrespective of whether the service request originated in IRIS or NEC.

6.Once the NEC users have extracted the data using OCR tool, into the webform, NEC users can now edit and enrich the data as required and save Intertrust entity admin users can now review the uploaded forms in PDF format and make changes directly on the PDF document using PDF editor (existing process), on email confirmation from the client (email is saved to Laserfiche as evidence)

7.The target contact information is shared with the identity KYC module. Note that in case of multiple owner-controllers for whom identity KYC needs to be performed, the same number of separate identity KYC requests are handed off to the identity KYC module

8.Risk assessment entity relationship KYC data will be downloadable at the click of a button for the purpose of manual entity relationship KYC process. The data can be downloaded from the NEC by either the entity admin team user or another authenticated Intertrust compliance user, performing entity relationship KYC, to use the data for the manual Risk assessment entity relationship KYC. If NEC is integrated with the new risk engine, the handoff of the entity relationship KYC data to the risk engine for entity relationship risk assessment, will be automatic and not manual

9.The incremental data collected for transaction profile, in addition to the rest of the data required for creation of the profile and captured as a part of the other sections already, will be automatically handed off to the Transaction Profile (TP) and Transaction Monitoring (TM) module to create the initial skeletal version of the transaction profile with minimal data.

10.Operational documents (if provided), which will be used eventually for entity administration purposes, will be saved in NEC till the time the entity is incorporated with the government. Once incorporated along with the rest of the necessary documents, they will also be manually saved into Laserfiche. The functional setup on operational documents (use case NEC_05) for the given jurisdiction and entity type will keep alerting the IRIS and NEC users by email and system notification till all expected operational documents are available and uploaded

 

1.Alternative Path 1

1.IRIS Authorized User saves incomplete form (i.e. incomplete data or documents related to relationship KYC, OR not all filing documents uploaded as against what is determined by jurisdiction and entity type) by clicking on save button and exits in IRIS system.

1.Post Conditions (Alternative path 1)

1.Incomplete data and documents are saved in IRIS

2.When the IRIS User accesses the form again, the IRIS User will be shown the section that was last saved by the IRIS User .

3.All the data and documents saved earlier will be displayed in the IRIS portal

4.No mention of the new service request as yet on NEC

 

1.Alternative Path 2

1.NEC User saves an incomplete form (i.e. incomplete data or documents related to relationship KYC, OR not all filing documents uploaded as against what is determined by jurisdiction and entity type) by clicking on save button and exits in NEC.

1.Post Conditions (Alternative path 2)

1.Incomplete data and documents are saved in NEC

2.When the NEC Authorized User accesses the form again, the NEC Authorized User will be shown the section that was last saved by the NEC Authorized User .

3.All the data and documents saved earlier will be displayed in the NEC system

4.No mention of the new service request as yet on IRIS
